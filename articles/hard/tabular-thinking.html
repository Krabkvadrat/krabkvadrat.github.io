Мышление аналитика

Табличное мышление

Табличное мышление - один из самых полезных навыков аналитика. Это умение позволяет понимать, что происходит с данными на каждом шаге: что содержится в одной строчке таблицы (какая гранулярность) и как она изменится после следующей операции.

Гранулярность таблицы

Прежде чем что-то делать с данными, задайте один вопрос: что является одной строкой в этой таблице?

Это и есть гранулярность (степень детализации данных: насколько мелко нарезана таблица). От неё зависит всё: какие операции допустимы, что означает агрегат, насколько правомерен JOIN.

ТаблицаГранулярность (одна строка = ?)Гранула
eventsОдно событие пользователяevent_id
usersОдин пользовательuser_id

Если не можете сформулировать гранулярность - вы ещё не поняли таблицу. Без этого любая трансформация будет наугад.

Нарисуй таблицу

Самый простой способ понять, что происходит с данными - нарисовать таблицу. Буквально 3-4 строки на бумаге или в заметках. Дальше любая операция становится очевидной. Разберём три базовые операции на маленькой таблице events.

WHERE

Убирает строки по условию. Гранулярность не меняется.

Было:

event_iduser_iddateevent
112024-01-01click
222024-01-01buy
312024-01-02click
432024-01-02click
 
SELECT *
FROM events
WHERE date = '2024-01-01'

Стало:

event_iduser_iddateevent
112024-01-01click
222024-01-01buy

Строк стало меньше, но одна строка по-прежнему означает одно событие.

LEFT JOIN

Добавляет колонки из правой таблицы. Все строки левой остаются, а если у одного ключа несколько совпадений справа - строки множатся.

users
user_idcity
1Москва
2Казань
3Самара
4Екатеринбург
events
event_iduser_iddateevent
112024-01-01click
212024-01-02buy
312024-01-03click
422024-01-01buy
532024-01-02click
632024-01-02share
 
SELECT users.user_id, city, date, event
FROM users
LEFT JOIN events ON users.user_id = events.user_id

Стало:

user_idcitydateevent
1Москва2024-01-01click
1Москва2024-01-02buy
1Москва2024-01-03click
2Казань2024-01-01buy
3Самара2024-01-02click
3Самара2024-01-02share
4ЕкатеринбургNULLNULL

Пользователь 1 был одной строкой, стал тремя. Пользователь 4 остался, но date и event NULL. Гранулярность поменялась: было "один пользователь", стало "одно событие пользователя".

GROUP BY

Сворачивает строки в группы. Гранулярность меняется.

Было:

event_iduser_iddateevent
112024-01-01click
212024-01-02buy
312024-01-03click
422024-01-01buy
532024-01-02click
632024-01-02share
 
SELECT user_id, COUNT(*) AS event_count
FROM events
GROUP BY user_id

Стало:

user_idevent_count
13
21
32

Каждый пользователь теперь одна строка. Гранулярность стала "один пользователь".

Рисуй с конца

Рисование работает в обе стороны. Когда задача непонятна - нарисуй не только входные данные, но и то как должен выглядеть результат. Потом думай как туда добраться.

Задача: найти второе по величине событие каждого пользователя.

Рисуешь что есть:

event_iduser_iddateamount
112024-01-01100
212024-01-02250
312024-01-0380
422024-01-01300
522024-01-02150

Рисуешь что хочешь получить - и понимаешь что нужно пронумеровать строки внутри каждого пользователя по убыванию суммы:

event_iduser_iddateamount№
212024-01-022501
112024-01-011002
312024-01-03803
422024-01-013001
522024-01-021502

Как только нарисовал столбец с номером - идёшь в Google или спрашиваешь LLM: "как пронумеровать строки внутри группы в SQL?" И узнаёшь про оконные функции и ROW_NUMBER. Потом фильтруешь WHERE № = 2 - и задача решена.

Визуализация желаемого результата подсказывает какой инструмент искать, даже если ты про него ещё не знал.

Чек-лист

Перед тем как писать запрос:

  • Что является одной строкой в исходной таблице?
  • Как выглядит результат который хочу получить - нарисовал?

В процессе:

  • Что станет со строками после следующей операции?
  • Гранулярность изменилась - я это учёл?

Если застрял:

  • Нарисовал желаемый результат и понял какой инструмент нужен?
  • Спросил LLM почему результат не совпал с ожиданием?
Analyst mindset

Tabular Thinking

Tabular thinking is one of the most useful skills for an analyst. It lets you understand what happens to data at every step: what is contained in a single row (its granularity) and how that will change after the next operation.

Table granularity

Before doing anything with data, ask one question: what does a single row in this table represent?

That is granularity (the level of detail: how finely the table is sliced). Everything depends on it: which operations are valid, what an aggregate means, whether a JOIN is appropriate.

TableGranularity (one row = ?)Grain
eventsOne user eventevent_id
usersOne useruser_id

If you cannot state the granularity, you have not yet understood the table. Without it, any transformation is a guess.

Draw the table

The simplest way to understand what is happening to data is to draw the table. Literally 3-4 rows on paper or in a note. After that, any operation becomes obvious. Let's walk through three basic operations on a small events table.

WHERE

Removes rows based on a condition. Granularity does not change.

Before:

event_iduser_iddateevent
112024-01-01click
222024-01-01buy
312024-01-02click
432024-01-02click
 
SELECT *
FROM events
WHERE date = '2024-01-01'

After:

event_iduser_iddateevent
112024-01-01click
222024-01-01buy

Fewer rows, but one row still means one event.

LEFT JOIN

Adds columns from the right table. All rows from the left table are kept, and if one key matches multiple rows on the right - rows multiply.

users
user_idcity
1Moscow
2Kazan
3Samara
4Yekaterinburg
events
event_iduser_iddateevent
112024-01-01click
212024-01-02buy
312024-01-03click
422024-01-01buy
532024-01-02click
632024-01-02share
 
SELECT users.user_id, city, date, event
FROM users
LEFT JOIN events ON users.user_id = events.user_id

After:

user_idcitydateevent
1Moscow2024-01-01click
1Moscow2024-01-02buy
1Moscow2024-01-03click
2Kazan2024-01-01buy
3Samara2024-01-02click
3Samara2024-01-02share
4YekaterinburgNULLNULL

User 1 was one row, now is three. User 4 remains but date and event are NULL. Granularity changed: was "one user", now "one user event".

GROUP BY

Collapses rows into groups. Granularity changes.

Before:

user_iddateevent
12024-01-01click
12024-01-02buy
12024-01-03click
22024-01-01buy
32024-01-02click
32024-01-02share
 
SELECT user_id, COUNT(*) AS event_count
FROM events
GROUP BY user_id

After:

user_idevent_count
13
21
32

Each user is now one row. Granularity became "one user".

Draw from the end

Drawing works both ways. When a task is unclear - draw not just the input, but also what the result should look like. Then think about how to get there.

Task: find the second largest event for each user.

Draw what you have:

event_iduser_iddateamount
112024-01-01100
212024-01-02250
312024-01-0380
422024-01-01300
522024-01-02150

Draw what you want to get - and you realise you need to number rows within each user by amount descending:

user_iddateamount#
12024-01-022501
12024-01-011002
12024-01-03803
22024-01-013001
22024-01-021502

Once you draw that numbered column, you search or ask an LLM: "how to number rows within a group in SQL?" And you discover window functions and ROW_NUMBER. Then filter WHERE # = 2 - task solved.

Visualizing the desired result tells you what tool to look for, even if you did not know it existed.

Checklist

Before writing a query:

  • What does a single row in the source table represent?
  • Did you draw what the result should look like?

While working:

  • What will happen to the rows after the next operation?
  • Did the granularity change - did I account for that?

If stuck:

  • Did you draw the desired result and figure out which tool you need?
  • Did you ask an LLM why the result did not match your expectation?
