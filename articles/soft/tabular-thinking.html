Мышление аналитика

Табличное мышление

Самый полезный навык аналитика — видеть данные в виде таблицы и понимать, как каждая операция меняет её форму. Это не врождённое чутьё: одни месяцами пишут SQL вслепую, другие уже через пару недель мысленно прослеживают каждую трансформацию. Хорошая новость — навык можно развить намеренно.

Зерно таблицы

Прежде чем что-то делать с данными, задайте один вопрос: что является одной строкой в этой таблице?

Это называется зерном (grain). От него зависит всё: какие операции допустимы, что означает агрегат, насколько правомерен JOIN.

ТаблицаЗерно (одна строка = ?)Ключ
eventsОдно событие пользователяevent_id
usersОдин пользовательuser_id
ordersОдин заказorder_id
daily_statsОдин день + один сегментdate + segment

Если вы не можете сформулировать зерно — вы ещё не поняли таблицу. Без этого любая трансформация будет наугад.

Что трансформации делают с таблицей

Каждый SQL-запрос или pandas-операция либо убирает строки, либо меняет зерно. Достаточно держать в голове одну схему:

ОперацияСтрокиЗерно
WHERE / .query()УбываютНе меняется
SELECT columns / [cols]Не меняютсяНе меняется
JOIN по уникальному ключу правой таблицыНе меняютсяНе меняется
GROUP BY / .groupby()Сворачиваются в группыМеняется
PIVOT / .pivot()СворачиваютсяМеняется

JOIN умножает строки, если ключ в правой таблице не уникальный. Всегда проверяйте это заранее.

Пример: шаг за шагом

Задача: посчитать среднее количество событий на пользователя в день. Зерно исходной таблицы — одно событие.

1
Фильтруем нужный период
Зерно: одно событие. Строк стало меньше, зерно не изменилось.
SELECT user_id, event_date, event_type
FROM events
WHERE event_date >= '2024-01-01'
2
Считаем события по пользователю и дню
Зерно меняется: теперь одна строка = пользователь + день.
SELECT user_id, event_date, COUNT(*) AS events_cnt
FROM events
WHERE event_date >= '2024-01-01'
GROUP BY user_id, event_date
3
Считаем среднее по дням
Зерно меняется ещё раз: одна строка = один день.
SELECT event_date, AVG(events_cnt) AS avg_events_per_user
FROM (
    SELECT user_id, event_date, COUNT(*) AS events_cnt
    FROM events
    WHERE event_date >= '2024-01-01'
    GROUP BY user_id, event_date
) t
GROUP BY event_date
ORDER BY event_date

Подзапрос здесь не случаен. После первого GROUP BY зерно изменилось — нельзя взять AVG(COUNT(*)) в одном проходе. Нужно сначала посчитать COUNT, зафиксировать новое зерно, а потом взять AVG. Подзапрос — это точка смены зерна.

Типичные ошибки без табличного мышления

JOIN без проверки уникальности

Если в правой таблице несколько строк на один ключ — строки тихо умножатся. Проверяйте заранее: SELECT key, COUNT(*) FROM t GROUP BY key HAVING COUNT(*) > 1

COUNT(*) после GROUP BY

После GROUP BY зерно изменилось. Если снова написать COUNT(*) — посчитаете группы, а не исходные события. Нужно чётко понимать, что находится в строке в каждый момент.

Агрегат по агрегату

AVG(conversion_rate) по уже посчитанным строкам — классическая ошибка. Среднее нужно считать по исходному зерну, а не по агрегату. Иначе маленькие группы получают тот же вес, что и большие.

Как развивать

1
Называйте зерно вслух
Перед любой трансформацией проговорите: «сейчас одна строка — это X». После: «теперь одна строка — это Y». Звучит банально, но убирает большинство ошибок.
2
Рисуйте мини-таблицы
Нарисуйте 3–5 строк «до» и 3–5 строк «после» операции. Бумага, заметки — неважно. Визуализация мгновенно показывает, где что-то пошло не так.
3
Считайте строки на каждом шаге
После каждой промежуточной трансформации добавьте SELECT COUNT(*) или len(df). Число строк — быстрая проверка того, что зерно ведёт себя ожидаемо.
4
Разбивайте запросы на шаги
Пишите подзапрос — проверьте его зерно — идите дальше. Монолитный запрос сложно отлаживать именно потому, что неясно, где и как изменилось зерно.
5
Объясняйте запрос без компьютера
Если можете описать словами, что делает каждый шаг и что будет в строке на каждом этапе — вы думаете таблицами. Если нет — вернитесь к шагу 1.

Главный вывод

Табличное мышление — это не знание синтаксиса SQL, а модель в голове. Аналитик, который её выработал, пишет правильные запросы с первой попытки не потому, что умнее, а потому что видит таблицу до того, как напишет код. Зерно → трансформация → новое зерно. Повторяйте это для каждого шага, и сложные запросы перестанут быть сложными.

Analyst mindset

Tabular Thinking

The most valuable skill an analyst can have is the ability to visualize data as a table and understand how each operation reshapes it. This is not some innate talent — some people write SQL blindly for months, while others track every transformation mentally within weeks. The good news: it can be developed deliberately.

The grain of a table

Before doing anything with data, ask one question: what does a single row in this table represent?

This is called the grain. Everything depends on it: which operations are valid, what an aggregate means, whether a JOIN is correct.

TableGrain (one row = ?)Key
eventsOne user eventevent_id
usersOne useruser_id
ordersOne orderorder_id
daily_statsOne day + one segmentdate + segment

If you can't articulate the grain, you don't yet understand the table. Without it, any transformation is a guess.

What transformations do to a table

Every SQL query or pandas operation either removes rows or changes the grain. One mental model covers all cases:

OperationRowsGrain
WHERE / .query()DecreaseUnchanged
SELECT columns / [cols]UnchangedUnchanged
JOIN on unique right-table keyUnchangedUnchanged
GROUP BY / .groupby()Collapse into groupsChanges
PIVOT / .pivot()CollapseChanges

JOIN multiplies rows if the right-table key is not unique. Always verify this upfront.

Example: step by step

Task: calculate the average number of events per user per day. The grain of the source table is one event.

1
Filter the date range
Grain: one event. Row count decreases, grain unchanged.
SELECT user_id, event_date, event_type
FROM events
WHERE event_date >= '2024-01-01'
2
Count events per user per day
Grain changes: one row now = one user + one day.
SELECT user_id, event_date, COUNT(*) AS events_cnt
FROM events
WHERE event_date >= '2024-01-01'
GROUP BY user_id, event_date
3
Average across users per day
Grain changes again: one row = one day.
SELECT event_date, AVG(events_cnt) AS avg_events_per_user
FROM (
    SELECT user_id, event_date, COUNT(*) AS events_cnt
    FROM events
    WHERE event_date >= '2024-01-01'
    GROUP BY user_id, event_date
) t
GROUP BY event_date
ORDER BY event_date

The subquery exists for a reason. After the first GROUP BY the grain changed — you can't do AVG(COUNT(*)) in one pass. Count first, lock in the new grain, then average. The subquery is a grain checkpoint.

Common mistakes without tabular thinking

JOIN without checking uniqueness

If the right table has multiple rows per key, the JOIN silently multiplies your rows. Always check: SELECT key, COUNT(*) FROM t GROUP BY key HAVING COUNT(*) > 1

COUNT(*) after GROUP BY

After GROUP BY the grain changed. Writing COUNT(*) again counts groups, not original events. You need to know what is in a row at every moment.

Aggregate of an aggregate

Averaging pre-computed conversion rates is a classic mistake. Averages must be computed on the original grain — otherwise small groups get the same weight as large ones.

How to develop it

1
Name the grain out loud
Before every transformation say: "right now one row is X." After: "now one row is Y." It sounds trivial but eliminates most mistakes.
2
Sketch mini-tables
Draw 3–5 rows "before" and 3–5 rows "after" the operation. Paper, sticky note, doesn't matter. Visualization instantly reveals where something went wrong.
3
Count rows at every step
After each intermediate transformation add SELECT COUNT(*) or len(df). Row count is a quick sanity check that the grain is behaving as expected.
4
Break queries into steps
Write a subquery, verify its grain, then go further. A monolithic query is hard to debug precisely because it's unclear where and how the grain changed.
5
Explain your query without a computer
If you can describe in plain words what every step does and what will be in a row at each stage — you're thinking in tables. If you can't — go back to step 1.

Bottom line

Tabular thinking is not knowledge of SQL syntax — it is a mental model. An analyst who has built this model writes correct queries on the first try not because they're smarter, but because they see the table before they write the code. Grain → transformation → new grain. Apply this to every step and complex queries stop feeling complex.
